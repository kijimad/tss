# Disk Driver + I/O Scheduler シミュレータ

ブラウザ上で動作するディスクドライバの実装。HDD の物理構造（プラッタ/ヘッド/シリンダ/セクタ）、DMA 転送、割り込み、バッファキャッシュ、4種類の I/O スケジューリングアルゴリズム(FIFO/SSTF/SCAN/C-SCAN)を可視化する。ヘッドの移動パスをリアルタイムで描画。

## 起動

```bash
npm install
npm run dev
```

## 画面の見方

- **上部**: ディスク種類(HDD/SSD)、スケジューラアルゴリズム選択、I/O 実行ボタン
- **左上**: **ヘッド移動パス** — Canvas 上でシリンダ軸（横）× 時間軸（縦）にヘッドの移動を描画
- **左下**: 統計（総シーク時間、平均シーク時間、リクエスト順序）
- **右**: I/O イベントログ（キャッシュヒット/ミス、リクエスト発行/完了、シーク/回転/転送時間）

## HDD の物理構造

```
          ← シリンダ (トラック) →
 ┌──────────────────────────────────┐
 │  ○○○○○○○○○○○○○○○○○○○○○○○○○○○○ │  プラッタ面 (ヘッド 0)
 │  ○○○○○○○○○○○○○○○○○○○○○○○○○○○○ │  プラッタ面 (ヘッド 1)
 │  ○○○○○○○○○○○○○○○○○○○○○○○○○○○○ │  プラッタ面 (ヘッド 2)
 │  ○○○○○○○○○○○○○○○○○○○○○○○○○○○○ │  プラッタ面 (ヘッド 3)
 └──────────────────────────────────┘
      ↑ ヘッド（アームで横移動 = シーク）
```

I/O 時間 = **シーク時間** + **回転待ち** + **転送時間**
- シーク: ヘッドを目的のシリンダに移動（0〜15ms、距離に比例）
- 回転待ち: セクタがヘッド下に来るまで（平均 4.17ms @7200RPM）
- 転送: データ読み書き（0.01ms/sector @150MB/s）

## I/O スケジューリングアルゴリズム

| アルゴリズム | 動作 | 利点 | 欠点 |
|------------|------|------|------|
| **FIFO** | 先着順 | 公平、実装簡単 | シーク最適化なし |
| **SSTF** | 最短シーク優先 | シーク時間最小 | 端のリクエストが飢餓 |
| **SCAN** | エレベーター（端で折り返し） | 公平＋シーク最適化 | 端付近のリクエストの待ち時間が不均一 |
| **C-SCAN** | 片方向のみサービス | 待ち時間が均一 | 折り返し時に無駄なシーク |

## 実験

### 実験 1: Random I/O でアルゴリズムを比較

1. HDD + FIFO を選択 → 「Random I/O」
2. ヘッド移動パスを確認（ジグザグに大きく動く）
3. アルゴリズムを SSTF に変更 → 「Random I/O」
4. ヘッドの移動がなめらかになり、総シーク時間が減る
5. SCAN → エレベーターのように端まで行って折り返す
6. C-SCAN → 一方向のみ進み、端で先頭に戻る

---

### 実験 2: 「Compare All Algorithms」

1. 「Compare All Algorithms」をクリック
2. 同じ LBA セットで FIFO/SSTF/SCAN/C-SCAN を比較
3. 表で各アルゴリズムの Total Seek / Avg Seek / Max Seek が見える
4. ★ マークが最も効率的なアルゴリズム
5. 通常は SSTF か SCAN が最良

---

### 実験 3: Sequential I/O

1. 「Sequential I/O」をクリック
2. 連続した LBA を読むので、ヘッドの移動がほぼ直線
3. どのアルゴリズムでもシーク時間がほぼゼロ
4. Sequential I/O が HDD で速い理由が分かる

---

### 実験 4: HDD vs SSD

1. HDD + FIFO で「Random I/O」→ シーク時間が大きい
2. SSD + FIFO で「Random I/O」→ シーク時間がほぼゼロ
3. 「Compare All Algorithms」で SSD を選択 → 全アルゴリズムの差がほぼない
4. SSD ではスケジューラの選択がほぼ意味をなさない（物理的なヘッド移動がないため）

---

### 実験 5: キャッシュの効果

右パネルのイベントログを見る:
- `CACHE MISS`: 初回アクセス → ディスク I/O 発生
- `CACHE HIT`: 2回目以降 → ディスク I/O なし（メモリから即返却）
- `EVICT`: キャッシュが満杯 → 最も古いエントリを追い出す

---

### 実験 6: SSTF の飢餓問題

1. SSTF を選択
2. 「Random I/O」を数回実行
3. 統計の「Request order (cylinder)」を見る
4. ヘッド位置に近いリクエストが優先され、遠いリクエストが後回しにされる
5. これが SSTF の「飢餓 (starvation)」問題

---

### 実験 7: SCAN（エレベーター）の動き

1. SCAN を選択 → 「Random I/O」
2. ヘッド移動パスを見る: 一方向に進み、端で折り返す
3. エレベーターと同じ動き（上に行って、全部拾って、下に戻る）

---

### 実験 8: I/O の時間内訳

右パネルの `DONE` イベント:
```
DONE #5 lba=1234 seek=3.2ms rot=4.2ms total=7.4ms
```
- `seek`: ヘッド移動（シリンダ距離に比例）
- `rot`: 回転待ち（HDD のみ、固定値 4.17ms）
- `total`: 合計

HDD の I/O で最も遅いのはシーク。だからスケジューラがシーク距離を最小化することが重要。

---

### 実験 9: DMA と割り込み

右パネルのハードウェアイベント（driver の内部）:
1. `SUBMIT`: リクエストがキューに投入される
2. ヘッドがシーク → 回転待ち
3. `DMA_START`: データ転送開始（CPU を介さずメモリに直接転送）
4. `DMA_COMPLETE`: 転送完了
5. `IRQ 14`: ハードウェア割り込みで CPU に完了を通知

---

### 実験 10: LBA と CHS の対応

LBA (Logical Block Address) は OS が使う連番。HDD 内部では CHS (Cylinder/Head/Sector) に変換される:
```
LBA 0   → C=0, H=0, S=1
LBA 63  → C=0, H=1, S=1  (次のヘッド)
LBA 252 → C=1, H=0, S=1  (次のシリンダ)
```
同じシリンダ内の LBA はシークなしでアクセスできる。

## アーキテクチャ

```
ユーザプロセス
  ↓ read()/write()
カーネル VFS
  ↓
ブロックデバイスドライバ  ← このシミュレータ
  ├── バッファキャッシュ (LRU, 32エントリ)
  │     ├── HIT → メモリから即返却
  │     └── MISS → I/O リクエスト発行
  ├── I/O スケジューラ (FIFO/SSTF/SCAN/C-SCAN)
  │     └── リクエストを並び替えてキューに
  └── ハードウェア制御
        ├── LBA → CHS 変換
        ├── シーク (ヘッド移動)
        ├── 回転待ち
        ├── DMA 転送
        └── IRQ 割り込み → 完了通知
  ↓
ディスクドライブ (プラッタ/ヘッド/セクタ)
```

## 実際のディスクドライバとの違い

| Linux kernel | このシミュレータ |
|-------------|-----------------|
| C 言語、数千行 | TypeScript 数百行 |
| /dev/sda, /dev/nvme0n1 | メモリ上の配列 |
| blk-mq (マルチキュー) | シングルキュー |
| deadline/mq-deadline/BFQ/none | FIFO/SSTF/SCAN/C-SCAN |
| struct bio / struct request | TypeScript オブジェクト |
| DMA コントローラ (実ハードウェア) | 関数呼び出し |
| IRQ ハンドラ | コールバック |
| ページキャッシュ (数GB) | LRU キャッシュ (32エントリ) |
